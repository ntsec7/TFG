\chapter{Especificación de Requisitos}

\section{Introducción}

\subsection{Propósito}

El propósito de esta especificación de requisitos es definir de manera sencilla, breve y clara las funciones y restricciones que deberá poseer el producto final que se va a desarrollar. \\

Para elaborarlo, se tendrán en cuenta las necesidades de los usuarios finales. El resultado final servirá como base para la construcción de la aplicación y se tendrá en cuenta a lo largo de todo su desarrollo. 

\subsection{Alcance}

La aplicación facilitará la toma de decisiones dentro de un grupo social, pudiendo usar opciones por defecto o personalizadas y eligiendo finalmente mediante diversos métodos de elección. A su vez, irá guardando y tomando los datos de la base de datos del sistema.

\subsection{Definiciones}

\begin{itemize}
    \item \textit{\uline{Administrador:}} Persona con acceso completo al sistema software y responsable del mantenimiento de la base de datos. De igual modo, podrá realizar las funciones de los \textit{clientes}.
    \item \textit{\uline{Cliente:}} Persona interesada en la toma de decisiones gamificada que ofrece la aplicación.
    \item \textit{\uline{Servidor:}} Equipo de cómputo en el que se implementará el sistema.
    \item \textit{\uline{Grupo:}} Conjunto de clientes que pueden tomar decisiones en común.
\end{itemize}

\section{Descripción general}

En este apartado, vamos a proporcionar una visión general del sistema. Con el objetivo, de determinar las funciones principales, datos necesarios y restricciones que afecten al desarrollo.

\subsection{Perspectiva del producto}
\begin{comment}
El sistema hará uso de una base de datos en la que se recogerá y guardará información relativa a los grupos y los distintos usuarios de la aplicación. \\

La interacción de los usuarios se llevará a cabo mediante una interfaz gráfica, la cual podrá ser accedida mediante una pantalla táctil.
\end{comment}

En la Figura \ref{fig:diagramapersproducto} se muestra la interacción entre los distintos componentes de la aplicación. Como podemos observar, la arquitectura se compone de tres elementos principales: el front-end desarrollado en Flutter, el back-end proporcionado por Supabase y el sistema de base de datos PostgreSQL. \\

El front-end se encarga de la interfaz de usuario con la que interactúan los usuarios a través de una pantalla táctil. Para ello, envía peticiones al back-end mediante solicitudes HTTPS en formato JSON. \\

El back-end gestiona la autenticación, la lógica de acceso a los datos y las funcionalidades de actualización en tiempo real. \\

Finalmente, los datos se almacenan en una base de datos PostgreSQL, donde se aplican políticas de seguridad a nivel de fila RLS (Row Level Security) para garantizar el acceso controlado a la información. \\

\subsection{Funciones del Producto}

Las funciones principales del sistema son la F1 y F2, las cuales vamos a comentar a continuación:

\begin{itemize}
    \item Código F1: Gestión de una base de datos (permitiendo insertar, actualizar y eliminar) en la que se almacenan los miembros de cada grupo, las opciones de cada decisión por defecto y la información de los usuarios y su rol en la aplicación.
    \item Código F2: Devolver información filtrada sobre ocio a través de APIs y/o \textit{scraping}.
\end{itemize}

Asimismo, la aplicación contará con las siguientes funciones:

\begin{itemize}
    \item Creación-Edición-Eliminación: Se usará tanto para usuarios como para grupos y opciones de las decisiones. 
    \item Añadir-Eliminar: Podrá utilizarse sobre los miembros de un grupo.
    \item Historial: Se podrán ver las votaciones anteriores, con la posibilidad de repetirlas.
\end{itemize}


\subsection{Características de los usuarios}

Es recomendable que los usuarios tengan conocimientos básicos de informática y estén familiarizados con el uso de aplicaciones en dispositivos móviles.

\subsection{Restricciones}

Las principales restricciones que debe cumplir el sistema son RES1 y RES2, las cuales vamos a comentar a continuación:

\begin{itemize}
    \item Código RES1: La aplicación estará disponible para dispositivos móviles,
    \item Código RES2: El sistema será multiplataforma, pudiendo ejecutarse en sistemas operativos Android e iOS.
    \item Código RES3: Siempre debe haber al menos un administrador creado en el sistema.
    \item Código RES4: Un administrador no podrá cambiar su propio rol o eliminarse.
    \item Código RES5: Será necesario disponer de conexión a Internet para poder utilizar la aplicación.
    \item Código RES6: No será necesario hardware adicional aparte del propio dispositivo móvil para utilizar la aplicación. 
\end{itemize}

\section{Requerimientos específicos}

\subsection{Requisitos funcionales}

\begin{itemize}
    \item \textit{\uline{Gestión de usuario:}}
        \begin{itemize}
            \item Código REQ1: Registro de usuario $\rightarrow$ El sistema permitirá al usuario crearse una cuenta personal asociada a él.
            \item Código REQ2: Modificación de usuario $\rightarrow$ El usuario podrá modificar los datos de su cuenta.
            \item Código REQ3: Eliminación de usuario $\rightarrow$ Un usuario podrá eliminar su cuenta personal.
            \item Código REQ4: Subida de imágenes $\rightarrow$ Los usuarios podrán adjuntar imágenes en el sistema.
            \item Código REQ5: Historial $\rightarrow$ El sistema permitirá a los usuarios consultar un histórico de las votaciones pasadas.
            \item 
        \end{itemize}
    \item \textit{\uline{Gestión de administrador:}} 
        \begin{itemize}
          \item Código REQ6: Creación de administrador $\rightarrow$ Para crear un administrador otro administrador deberá cambiar el rol de un usuario de cliente a administrador.
          \item Código REQ7: Eliminación de administrador $\rightarrow$ Para eliminar un administrador otro administrador deberá cambiarle el rol a cliente.
          \item Código REQ8: Registro de decisión $\rightarrow$ El administrador podrá registrar decisiones por defecto y guardarlas en la base de datos. Para ello, tendrá como mínimo que rellenar el ID de la decisión, el título y la lista de opciones.  
          \item Código REQ9: Gestión de datos $\rightarrow$ El administrador podrá modificar o eliminar la información almacenada en el sistema.
        \end{itemize}
    \item \textit{\uline{Gestión de los grupos:}} 
        \begin{itemize}
            \item Código REQ10: Creación de grupos $\rightarrow$ Los usuarios podrán crear grupos asignandoles un nombre y opcionalmente una imagen y lista de miembros.
            \item Código REQ11: Añadir usuarios $\rightarrow$ Los miembros de un grupo podrán añadir a otros miembros.  
            \item Código REQ12: Eliminar usuarios $\rightarrow$ Los miembros de un grupo podrán eliminar a otros miembros.
            \item Código REQ13: Salir de un grupo $\rightarrow$ Los usuarios que formen parte de un grupo podrán abandonarlo cuando deseen.
            \item Código REQ14: Permiso de voto $\rightarrow$ El sistema permitirá a los usuarios pertenecientes a un grupo participar en las votaciones activas del mismo.
        \end{itemize}  
    \item \textit{\uline{Gestión de las votaciones:}} 
        \begin{itemize}
          \item Código REQ15: Creación de la votación $\rightarrow$ El sistema permitirá crear una votación especificando título, tipo de votación y al menos dos opciones.
          \item Código REQ16: Eliminación de la votación $\rightarrow$ El sistema permitirá eliminar una votación activa.
          \item Código REQ17: Creación de opciones $\rightarrow$ El sistema permitirá que los usuarios añadan opciones a las decisiones, contando cada una con título y opcionalmente descripción e imagen.
          \item Código REQ18: Límite de tiempo $\rightarrow$ El creador de la votación podrá fijar un tiempo máximo para que el resto de miembros ejerza su derecho al voto.
          \item Código REQ19: Mecanismo de votación $\rightarrow$ El sistema permitirá al creador de la votación elegir el tipo de votación, pudiendo elegir entre una lista de opciones como ranking, consenso, pesos, etc.
          \item Código REQ20: Ganador de la votación $\rightarrow$ Una vez acabado el tiempo destinado a la votación o habiendo votado todos los miembros, el sistema calculará automáticamente el ganador según los votos y el método de votación elegido.
          \item Código REQ21: Mostrar los resultados $\rightarrow$ Al finalizar una votación los miembros podrán visualizar los resultados.
        \end{itemize}    
    \item \textit{\uline{Gestión de la aplicación:}}
        \begin{itemize}
            \item Código REQ22: Filtrado $\rightarrow$ El sistema contará con sistemas de filtrado y búsqueda mediante APIs o scraping para aportar opciones en la votación.
            \item Código REQ23: Inicio de sesión $\rightarrow$ Todos los usuarios entrarán al sistema mediante un inicio de sesión compuesto por nombre de usuario o correo electrónico y contraseña.
            \item Código REQ24: Cerrado de sesión $\rightarrow$ Todos los usuarios podrán cerrar su sesión del dispositivo que estén usando.
        \end{itemize}
\end{itemize}

\subsection{Requisitos no funcionales}

\begin{itemize}
    \item Código REQ25: Mantenibilidad $\rightarrow$ El código será estructurado y comentado para facilitar futuras modificaciones o ampliaciones.
    \item Código REQ26: Fiabilidad $\rightarrow$ El sistema deberá tener un funcionamiento estable, minimizando la aparición de errores durante su uso.
    \item Código REQ27: Robustez $\rightarrow$ El sistema deberá responder adecuadamente ante situaciones inesperadas o entradas incorrectas sin generar fallos en la aplicación.
    \item Código REQ28: Facilidad de uso $\rightarrow$ La interfaz será intuitiva, permitiendo a los usuarios usarla sin formación previa.
    \item Código REQ29: Protección de datos $\rightarrow$ Los datos de los usuarios serán protegidos y tratados siguiendo lo establecido en la normativa vigente de protección de datos(\href{https://www.boe.es/buscar/doc.php?id=DOUE-L-2016-80807}{Reglamento General de Protección de Datos (RGPD)} \cite{RGPD} y \href{https://www.boe.es/eli/es/lo/2018/12/05/3/con}{Ley Orgánica 3/2018 de Protección de Datos Personales y garantía de los derechos digitales (LOPDGDD)} \cite{LOPDGDD}).
    \item Código REQ30: Autenticación de usuarios $\rightarrow$ Únicamente podrán acceder al sistema los usuarios autenticados.
    \item Código REQ31: Compatibilidad multiplataforma $\rightarrow$ La aplicación deberá funcionar en sistemas operativos Android e iOS.
    \item Código REQ32: Integridad de la información $\rightarrow$ Los datos guardados no deberán perderse ni corromperse ante fallos del sistema.
    \item Código REQ33: Escalabilidad $\rightarrow$ El sistema deberá soportar el crecimiento del número de usuarios y grupos.
    \item Código REQ34: Concurrencia $\rightarrow$ El sistema deberá gestionar los votos emitidos simultáneamente por múltiples usuarios sin producir conflictos o pérdida de información. 
    \item Código REQ35: Sincronización $\rightarrow$ Los cambios producidos en una votación deberán ser transmitidos a todos los usuarios del grupo en un tiempo mínimo.
    \item Código REQ36: Consistencia de la información $\rightarrow$ Todos los usuarios deberán visualizar el mismo estado y resultado en las votaciones, evitando inconsistencias entre dispositivos.
\end{itemize}


\subsection{Requisitos de interfaces externas}

\begin{itemize}
    \item Código REQ35: Interfaces de usuario $\rightarrow$ Se podrá comunicar con el usuario para aprovechar los requisitos del sistema, el usuario indicará al sistema las operaciones que desea realizar e introducirá los datos que el sistema le pida.
    \item Código REQ36: Interfaces del software $\rightarrow$ La comunicación entre los módulos del sistema se realizará mediante bases de datos.
\end{itemize}
